# ADR-002: next-mdx-remote for MDX Content Processing

## Status

Accepted

## Date

2026-07-10

## Context

Posts are `.mdx` files authored in a `content/posts/` directory outside the Next.js `app/` tree. The blog needs to list all posts dynamically, extract frontmatter (title, date, description, tags), calculate reading time at build time, and render each post at `/blog/[slug]`. Two MDX processing strategies were evaluated.

## Decision

Use `next-mdx-remote` with `gray-matter` for frontmatter extraction. Posts live in `content/posts/*.mdx`. The `[slug]/page.tsx` route reads the file, parses frontmatter with `gray-matter`, and compiles the MDX body with `next-mdx-remote/rsc`. `generateStaticParams` enumerates all slugs at build time for static generation.

## Alternatives Considered

### Alternative 1: @next/mdx (built-in)

- **Description**: Native Next.js MDX integration. MDX files live inside the `app/` directory as route segments (e.g., `app/blog/my-post/page.mdx`)
- **Pros**: Zero extra dependency; official Next.js integration; no custom file-reading code
- **Cons**: Post files live inside `app/`, not a dedicated `content/` directory; no clean frontmatter API; listing all posts requires filesystem scanning from within `app/`; harder to separate content from application code
- **Why rejected**: The content/posts/ directory model is more maintainable and matches the author's mental model of "git push to publish." Keeping posts outside `app/` also makes the content portable in future phases.

## Consequences

### Positive

- Clear separation: `content/` is authoring space, `app/` is application code
- `gray-matter` gives typed frontmatter extraction with defaults
- `next-mdx-remote/rsc` works natively as a React Server Component — no client boundary needed for rendering
- Custom remark/rehype plugins (rehype-pretty-code) integrate cleanly in the compile step

### Negative

- Additional dependency (`next-mdx-remote`, `gray-matter`) vs zero extra for `@next/mdx`
- File reading uses Node.js `fs` — must stay in Server Components or build-time functions

### Risks

- `next-mdx-remote` major version upgrades have historically broken APIs. Pin to a minor version and test on upgrade.
