# ADR-003: Search Index as Public Static File

## Status

Accepted

## Date

2026-07-10

## Context

Edge search requires a queryable index of all post content and metadata. The index is derived entirely from `content/posts/*.mdx` at build time — it never changes between deploys. Three strategies for storing and accessing this index from the edge search route were evaluated.

## Decision

A build-time script serializes all post content and frontmatter into `public/search-index.json`. The edge search route at `/api/search` fetches this file via `fetch('/search-index.json')`, filters in memory, and returns ranked results. The file is served from Vercel's CDN edge cache and invalidated automatically on each deploy.

## Alternatives Considered

### Alternative 1: Bundle index into the edge function

- **Description**: Import the JSON directly in the route file — bundled into the function at build time
- **Pros**: Zero network hop to retrieve the index; fastest possible cold path
- **Cons**: Increases edge function bundle size by the full index size (50–300KB for a typical blog); approaching Vercel's edge function size limits as post count grows; no separate CDN caching
- **Why rejected**: Bundle size growth is unbounded as the blog grows. The CDN fetch is fast enough and keeps the function lean.

### Alternative 2: Store index in Turso at build time

- **Description**: Build script upserts the serialized index as a single row in the Turso DB; edge route reads it on each request
- **Pros**: Consistent storage location with other dynamic data
- **Cons**: Adds a DB read (with HTTP round-trip to Turso) to every search query; no benefit over a CDN-cached static file; increases Turso read usage
- **Why rejected**: Static data served from the CDN is faster and cheaper than a DB read. The index does not need transactional storage.

## Consequences

### Positive

- Index is CDN-cached at the edge — served from the same region as the search route
- Zero DB reads for search
- Build script is a simple Node script that runs as part of `next build`

### Negative

- Search results are stale until the next deploy (acceptable — posts are only added at deploy time anyway)
- The `public/` directory contains a generated file that should be in `.gitignore`

### Risks

- Large post libraries could produce a search index that is slow to transfer from CDN. Mitigation: compress the index (gzip, served automatically by Vercel) and cap indexed content per post at ~500 words.
